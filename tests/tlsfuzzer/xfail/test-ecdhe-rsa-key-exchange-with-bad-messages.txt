# TLS 1.2 ECDHE-RSA bad-message XFAIL (5 entries).
#
# Phase T90 originally XFAIL'd all 8 conversations of this script:
# every one failed early with "Server did not send extension(s):
# ec_point_formats", so none reached its actual bad-message check.
#
# Phase I101 fixed that — the TLS 1.2 ServerHello now echoes
# `ec_point_formats` (RFC 8422 §5.1.2) when the client offers it and
# an ECDHE suite is negotiated, and the server now honours the
# RFC 5246 §7.4.1.4.1 `{sha1,rsa}` default when the ClientHello omits
# `signature_algorithms`. With the handshake unblocked, 3 of the 8
# conversations now PASS (both `sanity` runs + `truncated ecdh_Yc
# value`).
#
# The remaining 5 are genuine TLS 1.2 bad-ClientKeyExchange handling
# gaps, surfaced now that the handshake gets far enough to exercise
# them:
#
# - `invalid point (...)` x4: when the client's ClientKeyExchange
#   carries an invalid ECDH point, the server aborts with
#   `internal_error`; RFC 4492 §5.4 / RFC 8422 expect
#   `illegal_parameter`. An alert-mapping gap in the TLS 1.2 ECDH
#   point-validation path — own follow-up phase.
# - `padded Client Key Exchange`: the server accepts a
#   ClientKeyExchange with trailing padding bytes instead of
#   rejecting the over-long message, and proceeds to ChangeCipherSpec.

invalid point (0, 0) :: internal_error

invalid point (self-inconsistent) :: internal_error
invalid point (self-inconsistent, y all zero) :: internal_error
