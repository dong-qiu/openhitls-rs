# Phase T91 — Finished mutation XFAILs (down from 72 to 6).
#
# T91 closed all of the empty / truncation / small-padding cases by
# tightening `decode_finished` to enforce RFC 8446 §4.4.4 ("verify_data
# is exactly Hash.length bytes") and by relaxing `get_body` so a
# header-only handshake message (`msg_data.len() == 4`) is no longer
# folded into "too short" but routed through the per-message decoder
# — which now produces a `decode_error`-mapping reason string.
#
# What's left: 6 conversations with `pad_right >= 131072` produce a
# handshake message larger than the 16384-byte maximum TLS record.
# tlslite-ng fragments the message across multiple records, but our
# server reads handshake messages one record at a time and rejects
# any record whose claimed inner length exceeds the record body —
# Step 6 in `tls13_server_do_handshake_body!` aborts with
# `"TLS 1.3 read key change not on record boundary"`. The server's
# alert (decode_error) then races tlslite-ng's still-streaming
# write, surfacing as "Unexpected message during send" on the
# tlsfuzzer side.
#
# Closing these requires real handshake-message reassembly across
# records (RFC 8446 §5.1 explicitly allows this, within a single
# key epoch). That's a non-trivial refactor of the handshake read
# loop. Scheduled for a future I- or T-phase.
#
